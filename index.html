In this section you are going to learn about one of the most convenient options that are available to generate embeddings for text, which is based on the SentenceTransformers framework.

Working with SentenceTransformers is the recommended path while you explore and become familiar with the use of embeddings, as the models that are available under this framework can be installed on your computer, perform reasonably well without a GPU, and are free to use.

Install SentenceTransformers

The SentenceTransformers framework is installed as a Python package. Make sure that your Python virtual environment is activated, and then run the following command on your terminal to install this framework:

pip install sentence-transformers

As always, any time you add new dependencies to your project it is a good idea to update your requirements file:

pip freeze > requirements.txt

Generating embeddings

With the model installed, you are now ready to generate an embedding. To do this, pass the source text to the model.encode() method:

embedding = model.encode('The quick brown fox jumps over the lazy dog')

The result is an array with all the numbers that make up the embedding. As you recall, the embeddings generated by the chosen model have 384 dimensions, so this is the length of the embedding array.

Hugging Face is an open-source hub dedicated to AI/ML models and tools. Seamlessly integrate Hugging Face and Elasticsearch by leveraging the transformers library or deploying available models directly into Elasticsearch.

This integration unlocks state-of-the-art transformer models for tasks like embeddings, classification, named-entity recognition, and more — all running natively against your indices.

Install the integration

Pull the latest eland package from pip and authenticate with your Hugging Face token:

pip install eland
huggingface-cli login

Deploy a model

Use the eland CLI to import a Hugging Face model into your Elasticsearch cluster:

eland_import_hub_model --url https://your-cluster.es.io \
  --hub-model-id sentence-transformers/all-MiniLM-L6-v2 \
  --task-type text_embedding

Once deployed, the model is available as an inference endpoint and can be used in ingest pipelines or directly in queries.

Level up your custom agents with tried-and-true prompts. This library is a working set of starting points used by Elastic engineers and customers when building security, observability, and search assistants. Each entry is a single, copyable prompt with the variables clearly marked.

Browse by category in the sidebar, copy what you need, and adapt it to your data and workflow. Prompts are written to be model-agnostic — they work with any modern instruction-following LLM, though the wording assumes the model has access to relevant context retrieved from Elasticsearch.

CISO Incident Summary

Create an executive summary of security incidents to communicate important details to a CISO. This prompt is designed for the moment an analyst needs to brief leadership without losing the technical signal.

Create an executive summary of the security incident below including:
- Severity level and business impact assessment
- Key findings from investigation
- Current incident status
- Resource requirements
- Strategic recommendations
- Proposed policy/control improvements
Format as a concise briefing optimized for executive review, highlighting critical decision points and required actions.

[Insert incident details here]

Pair this with retrieval over your incident-management index to pull in the relevant alert timeline, affected assets, and previous similar cases before invoking the model. A typical call returns a one-page brief that a CISO can read in under two minutes.

How prompts are structured

Every prompt in the library follows the same shape: a goal statement, a list of required outputs, a format instruction, and a placeholder block where the analyst pastes context. The placeholder pattern matters — it keeps the model's attention on the right substrate and makes the prompt easy to wire into a retrieval pipeline.

If you are building a custom agent on top of these prompts, you can replace the placeholder with a templated string that injects results from an Elasticsearch query. For example, a Python wrapper might look like this:

from elasticsearch import Elasticsearch

es = Elasticsearch("https://your-cluster.es.io")
hits = es.search(
    index="incidents",
    query={"match": {"incident_id": incident_id}}
)["hits"]["hits"]

context = "\n\n".join(h["_source"]["summary"] for h in hits)
filled_prompt = PROMPT_TEMPLATE.replace("[Insert incident details here]", context)

Once you have the filled prompt, send it to your LLM of choice. We recommend leaving the bullet structure in place — models follow the section ordering and produce a more consistent output.

Categories in this library

The full set covers a range of operational scenarios: analyzing a security case, asset inventory diagrams, custom Elasticsearch connector design, expanding evaluation datasets from seed indices, generating seed indices and test cases for evaluation, meeting summaries, negotiation assistance, security policy generation, technical discovery, and translating alert context into business-friendly language.

Each category exists because a customer or an internal team kept rewriting the same prompt by hand. If you build a prompt that you think belongs here, open a pull request against the public repository — contributions are welcome and reviewed weekly.

Variables and substitutions

Prompts use a small set of conventions for variables. Anything wrapped in square brackets is a substitution slot, like [Insert incident details here] or [user_question]. Anything in curly braces is a runtime variable filled by your application code, like {cluster_name} or {retrieval_results}. Keeping the two styles distinct helps when you are debugging a prompt that is not behaving — at a glance you can tell which substitutions are the analyst's responsibility and which are the system's.

Next steps

Pick a prompt that matches what you are building, copy it into your agent or application, and iterate. The prompts are intentionally short so that you can extend them with your own domain language without bumping into context window limits. If you want a worked example of wiring one into a retrieval-augmented chatbot, the Chatbot RAG reference app under the Apps section uses several of these prompts as its system messages.

This example shows how to load a Hugging Face embedding model into your Elasticsearch cluster so that it can be used directly inside ingest pipelines and queries. No external inference service required.

You will use the eland CLI to import the model, verify the deployment, and run a quick smoke test against an index.

Setup

Make sure you have Python 3.9+ available and that you can reach your Elasticsearch cluster from your machine. Then install eland and authenticate with Hugging Face:

pip install eland
huggingface-cli login

Load the model

Use the eland import command to push a model from the Hugging Face hub into your cluster:

eland_import_hub_model \
  --url https://your-cluster.es.io \
  --hub-model-id sentence-transformers/all-MiniLM-L6-v2 \
  --task-type text_embedding \
  --start

Once the command finishes the model is allocated and serving requests. You can confirm with a quick inference call:

POST _ml/trained_models/sentence-transformers__all-minilm-l6-v2/_infer
{"docs": [{"text_field": "Hello, Elasticsearch!"}]}

Next steps

Stateless — your new state of find with Elasticsearch

Discover this future of stateless Elasticsearch. Learn how we're investing in building a new fully cloud native architecture to push the boundaries of scale and speed.

Lorem
LL TB QH
By Leaf Lin, Tim Brooks, and Quin Hoxie On May 01, 2023

Where we started

The first version of Elasticsearch dates back to 2010. It was released as a distributed clusterable search engine allowing users to quickly search for and surface critical insights. Twelve years and over 80,000 commits later, Elasticsearch continues to provide users with battle-tested solutions to a wide variety of search problems. Thanks to the efforts of over 1,700 contributors, including hundreds of full-time Elastic employees, Elasticsearch has consistently evolved to meet the new challenges that arise in the field of search.

Early in Elasticsearch's life, even state-of-the-art servers were limited. Elasticsearch implemented a multiyear effort to render the cluster scalable and provide operators with a guarantee that any knowledge gained could safely be applied across cluster patterns and the busy worker setup.

Where we are going — future is stateless

One of the primary challenges when operating or orchestrating Elasticsearch is that it depends on numerous pieces of persistent state, it is therefore a stateful system. The three primary pieces are the translog, index store, and cluster metadata. This state means that storage must be persistent and cannot be lost during a node restart or replacement.

The existing Elasticsearch architecture on Elastic Cloud must duplicate indexing across multiple availability zones to provide redundancy in the case of outages. We intend to shift the persistence of this data from local disks to an object store, like AWS S3. By relying on external services for storing this data, we will remove the need for indexing replication, significantly reducing the hardware associated with ingestion. This architecture also provides very high durability guarantees because AWS S3, GCP Cloud Storage, and Azure Blob Storage replicate data to two availability zones.

Offloading index storage into an external service will also allow us to re-architect Elasticsearch by separating indexing and search responsibilities. Instead of having primary and replica instances handling both workloads, we intend to have an indexing tier and a search tier. Separating these workloads will allow them to be scaled independently and hardware selection to be more targeted for the respective use cases.

After undertaking a multi-month proof-of-concept and experimental phase, we are convinced that these object store services meet the requirements we envision for index storage and cluster metadata. Our testing and benchmarks indicate that these storage services can meet the high indexing needs of the largest clusters we have seen in Elastic Cloud. Additionally, backing the data in the object store reduces indexing costs and allows for simple tuning of the performance of search.

Benchmarking — 75% indexing throughput improvement

In order to validate this approach we implemented an extensive proof of concept where data was only indexed on a single node and replication was achieved through cloud object stores. We found that we could achieve a 75% indexing throughput improvement by removing the need to dedicate hardware to indexing replication. Additionally, the CPU cost associated with simply pulling data from the object store was much lower than indexing the data and writing it locally, as is necessary for the current architecture.

These performance tests were performed on a two node cluster against all three major public cloud providers (AWS, GCP, and Azure). We intend to continue to build out larger benchmarks as we pursue a production stateless implementation.

How modern data teams are reshaping the search experience

A look at how product, engineering, and customer-success teams across industries are using Elastic to unify their data platforms and accelerate decision-making.

Adipiscing
EM RS
By Elena Martinez and Ravi Sundar On May 01, 2026

The shift we are seeing

Across every industry we work with, the same pattern is emerging. Data is no longer siloed in isolated departments — it flows freely between product, engineering, customer-success, and leadership. The companies winning in this environment treat their data platform as a strategic asset, not a back-office cost center. Search, observability, and security all sit on the same foundation, and teams move faster because they no longer wait for someone in another department to pull a report.

What unites these organizations is a willingness to invest in tooling that scales with curiosity. When anyone in the company can ask a question of their data and get an answer in seconds, the rate of decision-making compounds. The traditional bottleneck of "let me ask the data team" disappears, and product teams ship more confidently because they have visibility into how customers actually use what they build.

Where the unified approach pays off

The most pronounced returns we see come from teams that have unified their operational data — logs, metrics, traces — with their business data in a single platform. Customer-facing search experiences benefit from the same vector models that power internal documentation discovery. Security teams reuse anomaly-detection pipelines that were originally built for performance monitoring. The boundaries between use cases blur, and the cost-per-insight drops accordingly.

One pattern that keeps emerging: the platforms that succeed are the ones that let teams move at their own pace. A marketing team using Elastic to power a content-recommendation engine doesn't need to know what the SRE team is doing with the same cluster. They share infrastructure, but their work is independent. This is the unlock — shared foundation, independent execution.

What to watch in the next 12 months

Three trends we are tracking closely. First, the consolidation of vector and lexical search into a single retrieval experience. The artificial separation between "semantic" and "keyword" approaches is rapidly disappearing, and platforms that handle both natively are pulling ahead. Second, the integration of generative AI into operational workflows — not as a chat interface bolted on the side, but as an analysis layer that surfaces patterns analysts would have missed. Third, the rise of platform engineering teams whose mandate is to make data infrastructure self-serve for the rest of the organization.

Each of these reinforces the others. A self-serve platform that handles both vector and lexical retrieval, with a generative layer that summarizes what the data is telling you, is the experience modern teams expect. It is also a non-trivial engineering challenge, which is why getting the foundation right matters more than ever.

Closing thought

The companies pulling ahead are not the ones with the most data — they are the ones with the shortest distance between a question and a useful answer. That distance is shrinking everywhere, and the teams that have invested in unified, self-serve data platforms are reaping the benefits. The next year will be about pushing that distance toward zero.
